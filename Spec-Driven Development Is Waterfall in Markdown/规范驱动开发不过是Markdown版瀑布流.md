# 规范驱动开发（Spec-Driven Development），不过是 Markdown 版瀑布流

SpecKit 拥有 77K 个 GitHub star。Scott Logic 实际测试后发现它慢了 10 倍。整个行业从 Confluence 身上什么都没学到。

---

SpecKit 有 77,000 个 GitHub star。AWS 围绕规范驱动开发构建了一整套 IDE。Tessl 凭借"规范而非代码才是事实来源"的承诺融资 1.25 亿美元。

这套说辞听起来很清晰：停止 vibe coding，写一份完整的规范，让 agent 去执行。工程师们喜欢这个。它看起来很严谨。感觉像是成年人终于入场了。

然后有人真正在一个真实项目上测试了它。慢了十倍。仪式更多。bug 一样不少。

整个行业围绕一个想法建立了完整的生态：如果我们给 AI agent 一份足够详细的规范，它们就能生产出可运行的软件。这和行业在外包、离岸开发、每一个试图用文档替代理解的模式上押下的赌注如出一辙。写得足够清楚，对面的某人（或某物）就能完美执行。

如果没人读 Confluence 页面，也没人会读你的 `.specify` 文件夹。而那个确实读了的 agent，恰恰缺失了文档里没有写到的一切。

我这么说，是因为我自己试过。在一个个人项目上，规范驱动开发感觉很有希望。独立环境，没有相互竞争的利益方，没有变动的需求。我写了规范，把 agent 指向它，输出结果还算合理。在这种孤立场景下，这个方法是有道理的。

然后我想了想，如果把它用在工作中会发生什么。那里的情况是：你对接的 API 在冲刺中途发布了一个破坏性变更；安全审计在规范提交后推翻了你的认证流程；客户根据你没被邀请的董事会会议重新排定了路线图优先级；事情不仅在迭代之间动态变化，在单个 ticket 的处理过程中就会变。

我意识到，规范驱动开发只在你其实不需要它的那个环境里才管用。当你是独自工作、需求稳定，规范不过是你和自己的一场对话。一旦加入团队、竞争优先级和真实世界的不确定性，规范在第一次站会之前就已经过时了。

---

## 价值 7.5 亿美元，押注在一个已解决的问题上

让我们具体说说 2025 年发生了什么。

GitHub 在 9 月发布了 SpecKit。它搭建起一个 `.specify` 目录，包含若干门控阶段：specify（规范）、plan（计划）、tasks（任务）、implement（实现）。它支持 23 个 AI agent 平台，还包含一个"宪法"文件，用于定义 agent 必须遵守的原则。工程师社区把它当作礼物来对待。

几周之内，SpecKit 就有了维基百科词条。Thoughtworks 将规范驱动开发纳入技术雷达（Technology Radar）。LinkedIn Learning 制作了课程。大会演讲随之出现。"规范驱动开发：终结 vibe coding"成了 DevLand 2026 上一个真实的演讲标题。

随后 AWS 推出了 Kiro，一个以规范驱动开发为核心范式的 VS Code 分支。规范驱动一切：带有验收标准的需求、设计文档和实现任务。超过 25 万名开发者在预览期使用了它。

然后是 Tessl，由前 Snyk CEO 创立。他们持有最极端的立场：代码是生成产物，规范才是被维护的来源。代码文件被标注为 `// GENERATED FROM SPEC - DO NOT EDIT`。他们以 7.5 亿美元估值融资 1.25 亿美元。公司有 21 名员工，仍处于测试阶段。

开源生态也爆炸式增长。BMAD-METHOD 模拟了 19 个命名 agent 角色：BA 的 Mary、PM 的 Preston、架构师 Winston、开发者 Devon。OpenSpec 采取极简 Markdown 路线。GSD 声称三个开发者能产出八个人的工作量。到 2026 年 3 月，这一领域已有超过 30 个框架被梳理出来。

叙事是统一的：vibe coding 是病，规范是药。

只是这药不管用。

## 真正测试时会发生什么

Scott Logic 的工程师在真实代码库上跑了 SpecKit——不是待办应用，不是周末项目，而是有真实约束的生产系统。仅规范阶段就生成了 2,500 行 Markdown，其中包含一份 406 行的"研究文档"，告诉他们一些他们对自己技术栈早已了解的事情。实现结果仍然有 bug。整个过程花费的时间大约是迭代式提示（iterative prompting）的 10 倍。他们的结论是："我认为这不是一个可行的流程，至少在其最纯粹的形式下不是。"

Fowler 和 Bockeler 的分析对 Kiro 得出了类似结论。为了修一个小 bug，Kiro 生成了四个用户故事和十六条验收标准。他们将其描述为"用大锤砸坚果"。更糟的是，agent 完全忽略了规范中的部分内容，并在明确指示不要这样做的情况下仍然生成了重复代码。

Augment Engineer 记录了一个团队为了在页面上显示一个日期而产出了 1,300 行 Markdown 的案例。他们的判断是："我们重新发明了前期大规模设计（Big Design Up Front）。只是把 Word 文档换成了 Markdown，把项目经理换成了 LLM。"

Marmelab 的工程团队独立得出了同样的结论。他们的博客标题说明了一切："瀑布流的反击。"规范驱动开发试图把开发者从软件开发中移除。这和瀑布流犯下的错误一样，也因同样的原因失败了。最重要的信息在构建过程中才会浮现，而不是在此之前。

行为驱动开发（Behavior-Driven Development）的先驱 Gojko Adzic 也发表了看法。他称 SDD 是"瀑布的复仇，或者说 BDD 发展到了新的极端"。

而 SpecKit 本身呢？开发者已经开始质疑这个项目是否还在积极维护。各项功能正在悄然被吸收进 GitHub Copilot 的原生计划模式（plan mode）中。那个引发这场运动的工具，可能正在变成它本该补充的产品中的一个功能。

## 为何规范作为 Agent 输入会失败

SDD 社区对这些失败的反应是可以预料的：规范写得还不够好。要写更好的规范，更详细，加更多验收标准，给 agent 更多上下文。

这完全没抓住重点。规范用于人与人之间的沟通是没问题的——大家共享上下文，可以追问，可以读出言外之意。问题在于把规范作为人与 AI agent 之间的接口。这就是这个模型失效的地方。

AI agent 精确地执行规范里写的内容。它无法问"设计师同意了这个流程吗？"它无法标记"这和 DevOps 说的关于部署管道的内容有冲突。"它无法感知到范围在规范提交后四十分钟的一条 Slack 消息里变了。它把文档当作字面意思来看，生产出匹配文字、而非匹配意图的代码。

你实际上并非在一份规范上工作。你在处理一个需要经过设计师、DevOps 工程师、产品经理和利益相关方讨论才能落地的需求。这些人中的每一个，对"完成"意味着什么都有着不同的心智模型。设计师从用户流程出发，DevOps 从部署约束出发，产品从业务结果出发。你的规范把所有这些视角压扁成一种声音：你的声音。然后你把这份被压扁的文档交给 agent，期待它产出整个团队都能交付的东西。

即便你写出了所能想象到的最完整的规范并提交到仓库，你也不会期望你的设计师、DevOps 工程师和产品经理打开一个 `.specify` 文件夹，审阅你准备喂给 agent 的内容。他们有自己的工具、自己的工作流、自己的上下文。规范成了你和 LLM 之间没有其他任何人签署的合同。

然后还有规范腐化（spec rot）的问题。在传统开发中，一份过时的规范令人恼火。在规范驱动的 agent 开发中，一份过时的规范是危险的。agent 会自信地根据过时的需求去执行。它不会标记漂移，不会问是否有变化，它会生成匹配一份已不再匹配现实的文档的代码。而且它会很快地做到这一点。

这和敏捷时代把全面文档做死是同一个失败。我们学到了这个教训，我们把它写了下来。显然，没有人读那份文档。

## 现实中什么有效

问题不在于 AI agent 是否需要结构——它们确实需要。vibe coding 是混乱的。问题在于这个结构从哪里来。

规范驱动开发的答案是：先把结构写下来，再把它交给 agent。但真实项目是动态的。需求在一个 ticket 的处理过程中就会变化，范围在一次迭代内就会偏移。周二的客户电话重塑了你周一写好规范的功能。周四浮现出的一个部署约束推翻了你周三喂给 agent 的架构。SDD 假设稳定性。真实团队在持续运动中运作。

能处理这种现实的工作流，从对话而非文档中给 agent 提供结构。

开同步会。把设计、DevOps、产品，以及所有对需求有上下文的人叫进来。告诉他们你会录音。在电话里尽可能多地收集信息。设计师说"那个流程对屏幕阅读器来说会出问题"。DevOps 负责人说"我们不能在 canary 配置后面部署那个"。产品基于销售团队昨天反馈的内容改变了范围。这一切都重要。这一切都不会出现在规范里，因为写规范的人不会知道需要把它们包含进去。

电话结束后，拿着会议记录，用 LLM 把所有内容整理成可读格式：已做决策、已识别约束、团队达成一致的验收标准、标记出的待解问题。把这些提交到代码库旁边。不是作为规范，而是作为团队实际对齐内容的结构化记录。

然后分配工作，放到团队使用的任何看板上。每个 ticket 都填入明确的说明、验收标准和完成定义（definition of done），直接从结构化会议记录中提取。

没有人会打开一份 40 页的规范文档。每个人都会读自己分配的 ticket。

这就是"规范"真正存在的地方：工程师本来就会阅读的自然工作单元里。你没有移除文档，你只是把它移到了人们真正会看的地方。

对于每个 ticket，自己写 prompt。在 ticket 描述旁边说明需要做什么。使用工具：用 Superpowers 做实现规划，用 Serena 做具有代码库感知的编辑，以及栈所需的任何 MCP server。prompt 就是规范，范围限定在单个任务，由在做决策时在场的人来写。

特别要提示模型：每当一个决策被做出，或者某个部分的实现完成时，都要记录下来。每一个权衡取舍，每一个相对于备选方案的选择。不是作为规范，而是作为追踪记录（trace）。

然后你会开始注意到：事情按照实际讨论的内容运转，而不是按照某一个人对需求的想象。当事情在迭代中途发生变化——它们总会变化——追踪记录捕捉到了这次转向。规范只会是错的。

## 追踪记录改变了一切

当事故发生时，你不只是有代码可以调试，你还有一份关于发生了什么、以及为什么可能存在这个问题的完整追踪记录。

团队决定采用某种方案时的那场会议。指定验收标准的那个 ticket。实现功能的那条 prompt。解释为何选择某种方案而非另一种的决策日志。每一步都可以追溯到一次人类对话，而非一份在第一行代码写下之前就已过时的生成文档。

想想你今天是怎么调试的。出了问题，你打开规范，和实现做比较，试图弄清楚它们在哪里发生了偏离。但那份规范是两周前写的，在范围变化之前，在三次不同对话重塑了需求之前。你在拿一份幻想文档和现实做比较，然后想搞清楚哪个在撒谎。

有了决策追踪记录，你沿着线索走下去。结构化会议记录告诉你团队达成一致的内容，ticket 告诉你验收标准，决策日志告诉你实现过程中做出了哪些权衡、原因是什么。你可以把 bug 追溯到那个出了问题的具体假设，还能看到那个假设是在哪场对话中形成的。

规范只会是静默地错误。追踪记录讲述的是代码如何演化以及为什么演化的故事。

这就是在保持自身投入的同时拥抱 AI 的含义。当你写了一份规范并把它交给 agent，你把自己从流程中移除了。当代码交回来时，你是在审阅别人的工作，你是审阅者，不是构建者。

当你参与会议、写 prompt、捕捉决策时，你嵌入在工作中。你在有上下文的情况下与 AI 协作，带着这段代码为何存在、它应该做什么的完整历史。你在流程之中，而不是从外部旁观。

## Confluence 的回响

SpecKit 有 77K 个 star，Kiro 吸引了 25 万名开发者，Tessl 拿到了 7.5 亿美元，不到一年内 30 多个框架被梳理出来，大会演讲，维基百科词条，整套仪式。

而他们解决的，是 Confluence 在 2005 年就解决过的同一个问题：给工程师一个结构良好的地方，用来写没人会读的东西。

SDD 工具永远无法修复的是：某一个人写下来的内容，与一个跨职能团队实际需要交付的内容之间的鸿沟。再多结构化的 Markdown 也弥合不了这个鸿沟。规范永远只是某一个人的理解，而工作需要所有人共同参与。

讽刺的是，AI 在这个流程中真的很有用。不是作为你规范的读取者，而是作为你对话内容的结构化工具、你决策的记录者、你推理过程的追踪器。AI 在你工作流中最好的角色，不是替代人与人对齐的那一步，而是让那一步的产出变得可检索、有结构、永久保存。

规范驱动开发运动提出了一个正确的问题：我们如何给 AI agent 提供它们产出有用代码所需的结构？他们只是给出了错误的答案。结构不来自于某一个人在工作开始前写下的文档，它来自于那份文档本应替代的那些对话。

你的 agent 不需要规范，它需要的是只有你在场时才能获得的上下文。

**规范是你和 LLM 之间没有其他任何人签署的合同。**
