# Claude Code 不再是编程助手

> 作者：Alireza Rezvani（Reza）
> 发布日期：2026年4月1日
> 原文链接：[Medium](https://medium.com/@reza.rezvani)

---

*注：AI 工具协助了本文的编辑工作。文中的观察、经历和问题完全来自作者本人。*

上周三，我查看了终端历史记录，数了一遍。47 条命令。没有一条是带着我自己写的代码的 `git commit`，没有一条是为我亲手编写的内容执行的构建命令。每一条命令，要么是 Claude Code 的斜杠命令（slash command），要么是对 CLAUDE.md 文件的配置修改，要么是对某个在我处理其他事情时运行的智能体（agent）产出的审查。

*开发者在终端前查看智能体仪表盘而非编写代码 | 图片生成工具：Gemini Pro © Alireza Rezvani*

那天我上线了三个功能。我的工程师团队合并了两个由我设计架构的 pull request。一份竞争情报报告在我完成早晨的日常工作流程之前就已经发到了我们的通讯频道。而我一行生产代码都没写。

我是一名 CTO。写代码曾经是我一天中最不愿被打扰的时段。现在我不确定自己是什么时候停止守护它的——也不确定那是我主动选择的，还是这个工具替我做了选择。

## 我最初使用 Claude Code 时的样子

我从 2025 年 1 月/2 月（第一个版本发布时）就开始每天使用 Claude Code。那时的心智模型很简单：一个住在终端里的结对编程（pair programming）伙伴。我描述我想要什么，Claude 建议代码，我审查、接受或修改，然后继续。人主导，AI 跟随。

这套节奏很熟悉。就像自动补全加上了更好的判断力——一个能对话的 Copilot。我围绕这套工作流建立了肌肉记忆：提问、审查、应用。提问、审查、应用。这段关系有清晰的形态：我是开发者，Claude 是助手。

"助手"这个词的分量超过了我当时的意识。它定义了权力关系：我设定方向，我做架构决策，我决定什么时候算完成。Claude 加速了机械性的部分，但创造性和战略性的部分是我的。

我不记得那种状态具体是哪周停止的。

## 悄然改变这段关系的功能

这种转变不是在某个单一版本中发生的。它跨越了一系列细小的能力迭代而积累，每一次都把这段关系向"结对编程"更远处推移，推向一个我当时还没有名字的东西。

Skills 在 2025 年 10 月上线。那是我第一次**为 Claude 写指令**，而不是**与 Claude 一起写代码**。

SKILL.md 文件不是一条提示词（prompt）。它是一套标准作业程序（standard operating procedure）。它告诉 Claude 如何每次以同样的方式执行一项任务，不需要我在场。

当我写下第一个 skill——一个为我们营销团队生成内容简报的工具——时，我并没有把它视作委托授权。但它恰恰就是。我在把我的判断力编码进一份文件，由 AI 独立执行。我现在在一个公开代码库里维护着 200+ 个自定义 skill。每一个都代表着一个我过去需要手动做的决定。

Hooks 在 2025 年 9 月跟进。钩子（hook）是一段在生命周期事件触发时自动运行的脚本——在 Claude 写入文件后、执行命令前、会话结束时。

我设置了一个钩子来对照我们的品牌规范校验邮件营销内容。然后我就停止手动审查那些内容了。不是因为我决定停止，而是那个钩子在捕获所有我本会捕获的问题。我的注意力转移到了别处。质量关卡已经在那个特定环节取代了我。

无头模式（headless mode）改变了时间边界。在无头模式出现之前，Claude 工作，我才工作。

我们是同步的。后来我写了一个 bash 脚本，每周日晚上运行 Claude，生成我们的每周分析报告。

周一早晨，报告已经在我们的通讯频道里等着了。Claude 在夜间工作了，没有我。结对编程这个比喻彻底崩了——你不可能和一个在你不在场时独自通宵工作的东西结对编程。

Agent Teams 在 2026 年 2 月移除了我作为通讯瓶颈的角色。有子智能体（subagent）时，Claude 向我汇报——我还是枢纽。有了 Agent Teams，多个 Claude 实例通过共享任务列表相互协调。

它们互发消息，认领任务，解决依赖关系。我不再需要在它们之间路由信息，我只是在审查它们的集体产出。正是这一点促使我构建了 OpenClaw——一个自托管的编排平台（orchestration platform）——因为就连 Anthropic 的原生工具也无法支撑我的团队在规模化场景下所需的协调模式。

Computer Use 和 /loop 在 2026 年 3 月让这种转变无可辩驳。Claude 现在可以看到我的屏幕、点击界面，并在我的笔记本电脑关盖时在云端基础设施上运行计划任务。这个工具不再需要我在场，不再需要我打开终端，不再需要我的注意力——直到它有成果要向我展示。

每一个功能在发布时都被庆贺为生产力的提升。它们确实也是。但那些逐步减少人类参与度的生产力提升，有一种积累效应，而这种效应从来不出现在发版日志里。

## 我意识到自己在管理而非编码的那天

三月中旬的一个周四。我花了九十分钟为一个新项目编写 CLAUDE.md 文件。不是在写代码——是在写文字。仔细措辞，关于我们偏好的错误处理模式，定义命名约定，描述我希望 Claude 遵循的架构原则，说明哪些文件禁止修改以及原因。

写到一半，我停下来了。

我以前写过这份文件。不是为 Claude 写的——是为两年前我入职的一名初级开发者写的。结构相同，具体程度相同。

目的相同——把期望定义得足够清晰，让对方能够在没有持续监督的情况下独立工作。

CLAUDE.md 是入职文档（onboarding document）。Skills 是标准作业程序。Hooks 是自动化代码审查。斜杠命令是委托接口（delegation interface）。Auto-memory 等价于新团队成员逐渐了解你的偏好、自动调整而不需要被提醒。

我不知不觉建立起了一整套管理层。我日常工作的产物不再是代码，而是配置、指令和审查。

我把这件事讲给一个朋友——也是 CTO，也深度使用 Claude Code——她说了一句话，让我一直在回味："我以前是批量生产代码的，现在我是批量生产上下文（context）的。"她说得对。稀缺资源不再是写代码的能力。

在我们看来——**稀缺的是判断力、品味，以及告诉代码该成为什么样子的组织知识（institutional knowledge）。**

## 数字背后的真实情况

我想说具体一点，因为"提高了生产力"这种模糊说法是每一篇工具发布博客的标配，而这篇文章不是那种东西。

在三周的跟踪期内，以下是我对我七人工程团队的实测数据：

- 代码审查周转时间：4.2 小时 → 38 分钟（钩子 + 自动化质量关卡）
- 每周分析报告：6 小时手动工作 → 12 分钟审查（周日夜间无头运行）
- 功能脚手架（feature scaffolding）：每个功能 3~4 小时 → 45 分钟 CLAUDE.md 精调 + 审查
- 跨会话协调：以前没有开会就无法实现 → Agent Teams 异步处理

对于一个按混合费率计算的七人团队，这些节省下来的小时数，每月大约相当于 23,000 美元的工程产能被重新定向。不是节省——没有人被裁员。而是时间从机械性执行转移到了架构思考、客户沟通和战略规划上。

一致性的收益同样重要。我的 CLAUDE.md 文件在执行标准，而这些标准过去是我通过 pull request 审查来执行的。这些标准不会在周五下午松懈，不会因审查者不同而有差异，它们以人工审查永远无法做到的方式保持确定性。

而规模效应是真实的。一个人同时编排五个 Claude Code 会话，产出的可用成果比一个人独自编写代码要多。我测量过，数字是清楚的。

但有些问题，数字无法回答。

## 我不确定我们还保留着什么

我还能像过去亲手写每一行代码时那样理解自己的代码库吗？上周我打开了一个 Claude 跨三个会话生成并重构的文件。它运行得很完美，测试通过，架构干净。我读了十分钟才弄清楚它在做什么。这在我自己写的代码上以前从未发生过。

精心设计代码的满足感，正在被精心设计提示词（prompt）的满足感所取代——这两者等价吗？当一份写得好的 CLAUDE.md 持续产出高质量的输出时，我确实会有真实的成就感。但那是一种不同于解决一道困难算法题的感觉。我不知道它是更差、还是只是不同、还是我在浪漫化一些我应该放手的东西。

当我调试的是智能体的输出而非代码时，我是在培养一种新技能，还是在失去一种旧技能？大概两者都有。但我注意到我的调试直觉在改变。我过去是顺着逻辑追溯，现在是顺着指令追溯——是哪条 CLAUDE.md 语句产生了这个行为，哪个 skill 被触发了，哪个钩子执行了或者没有执行。这种认知技能是不同的。

它是更好还是更差，取决于你认为一名 CTO 的时间应该花在什么地方。

如果 Claude Code 负责实现，我的实际价值是什么？我认为答案是战略、架构和品味。但我足够诚实地承认，我过去以为我的价值在于代码本身，而我对那一点的判断是错的——所以我也可能对这一点的判断同样有误。

我的一位高级工程师在上次回顾会议上说了一句话，我一直在反复琢磨："我觉得我正在一份一年前根本不存在的工作上越做越好，同时在我接受训练时的那份工作上越来越生疏。"他不是在抱怨，他只是在观察。我没有一个好的回答，现在也还没有。

## 没有人画出来的那条轨迹

这里有一个模式，延伸到了远超 Claude Code 的范围之外。

文本编辑器，然后是 IDE，然后是自动补全，然后是 Copilot，然后是智能体，现在是编排平台。

每一步都把开发者推离原材料——代码本身。每一步都被框架为赋能："更快写代码"、"专注于难的部分"、"让 AI 处理模板代码"。每一种框架都是准确的。但累积的方向是明显的：人正在从生产者移向监管者（supervisor）。

我不是说这是坏事，我是说它发生得比任何人预测的都要快，而我认识的大多数开发者还没有给它命名。他们感受到了——那种模糊的感觉，觉得自己的日常工作变了，感觉自己在做与所受训练不同的事情——但他们把这归因于具体工具，而不是更广泛的趋势。

我构建了 OpenClaw——一个自 1 月以来就在生产环境运行的自托管智能体编排平台——这件事本身就是这种转变的证据。我没有构建一个更好的代码编辑器，没有构建一个更智能的自动补全，我构建的是一套 AI 智能体的管理层。这是我的技术直觉告诉我缺少的那块拼图。不是更多代码生成，而是更多编排。

当我看 2026 年 3 月——Computer Use、用于计划任务的 /loop、远程控制、百万 token 的上下文窗口——我看到的不是功能特性，我看到的是一个世界的基础设施，在那个世界里，开发者的主要工作不是写代码，而是把意图（intent）定义得足够清晰，使得自主系统能够执行它。

Anthropic 自己的《2026 年智能体编程趋势报告（Agentic Coding Trends Report）》印证了这个方向：95% 的专业开发者现在每周至少使用一次 AI 编程工具，75% 的人依赖 AI 完成至少一半的工程工作。那不是早期采用者的数字，那是一个职业正在转型的数字。

## 我没有一个整洁的结论

这是我应该告诉你这意味着什么、你应该怎么做的地方。我没有。

我有的是一个观察：在 14 个月的 Claude Code 使用中，我的工作变化比此前 14 年的软件工程生涯更大。而这种变化并不戏剧化，它是渐进的，功能一个接一个，会话一次接一次，直到某个周三我看着终端历史，意识到我一整天没写代码——而那是这个季度我最高产的一天之一。

我每天早晨还是会打开终端，我还是自称 CTO。但终端不再是我写代码的地方，而是我分派工作、审查输出、优化那些塑造智能体产出的指令的地方。

我那位高级工程师的观察一直萦绕着我：在一份一年前不存在的工作上越做越好，在他接受训练时的那份工作上越来越生疏。我不认为这是一种失败，但我也不确定它算不算成功。
