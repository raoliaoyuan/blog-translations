# 专业技术文章英译中翻译规范（Translation Skills）

本文档定义了将英文技术文章翻译为中文时应遵循的风格、术语处理和格式规范。适用于面向有技术背景读者的深度技术分析、架构解读、工程实践类文章。

---

## 一、总体原则

1. **忠实原文**：严格遵循原文的论述结构、段落划分和逻辑链条，不添加原文没有的观点或内容。
2. **语言克制**：使用简洁、准确、专业的中文表达。不使用不必要的比喻、夸张、感叹或情绪化修辞。
3. **读者定位**：目标读者是有一定技术基础的工程师和技术管理者，不需要科普性解释。
4. **可读性优先**：在忠实原文的前提下，按照中文的自然语序调整句式，避免翻译腔。长句应适当拆分。

---

## 二、术语处理

### 2.1 专业术语标注

所有专业技术术语在**首次出现**时，必须同时给出中文翻译和英文原文，格式为：

```
中文翻译（English Original）
```

**示例：**

- 自愈查询循环（self-healing query loop）
- 死代码消除（dead-code elimination）
- 延迟发现机制（deferred discovery）
- 编译时常量（compile-time constant）
- 上下文窗口（context window）
- 运行时门控（runtime gating）

### 2.2 后续引用

同一术语在首次标注后，后续出现时可以只使用中文或英文，保持上下文自然流畅即可。如果间隔段落较远（超过 3 个大段落），可再次标注一次。

### 2.3 不翻译的内容

以下内容保留英文原文，不做翻译：

- **代码标识符**：变量名、函数名、类名、方法名等。例：`isConcurrencySafe()`、`autoDream`、`MEMORY.md`
- **命令和路径**：`npm pack --dry-run`、`.npmignore`、`*.map`
- **产品/项目专有名称**：Claude Code、GrowthBook、Hacker News、npm、Bun
- **缩写与行业惯例**：API、CLI、ARR、token、schema、I/O、A/B 测试、harness

### 2.4 可灵活处理的术语

某些术语已有被广泛接受的中文表述，可直接使用中文而不必每次附注英文：

- model → 模型
- streaming → 流式传输/流式
- cache → 缓存
- bug → bug（已约定俗成的外来词）
- token → token（AI 领域已约定俗成）

## 三、格式与排版

### 3.1 文档结构

- 保留原文的标题层级（h1/h2/h3），使用对应的 Markdown 标记（`#`/`##`/`###`）。
- 原文的分隔线（thematic break）使用 `---` 保留。
- 列表（有序/无序）保持原文结构不变。

### 3.2 元信息区

文章开头应包含以下元信息：

```markdown
> 作者：[原作者姓名]
> 发布日期：[原文日期]
> 原文链接：[链接]
```

### 3.3 代码与命令

- 行内代码使用反引号：`` `feature('KAIROS')` ``
- 代码块使用三反引号并标注语言类型
- 终端命令直接保留原文，不翻译

### 3.4 加粗与强调

- 保留原文的加粗（`**bold**`）和斜体（`*italic*`）标记位置
- 如原文通过加粗强调某个关键论点，翻译后同样加粗

---

## 四、语言风格

### 4.1 应做

- 使用短句、主动句式
- 技术描述直接、精确
- 保留原文作者的论证节奏和语气轻重
- 原文如有幽默或讽刺，用自然的中文还原意味，不过度渲染

**示例（好的翻译）：**

> 原文：Strip that away and you have an expensive autocomplete.
>
> 译文：去掉这些，你得到的只是一个昂贵的自动补全工具。

### 4.2 不应做

- ❌ 添加原文没有的感叹号或语气词
- ❌ 使用网络流行语或口语化表达（如"牛逼"、"干货满满"、"炸裂"）
- ❌ 对原文本身简洁的表述进行冗余展开或解释
- ❌ 使用堆砌同义词的方式强调（如"极其重要且关键且不可或缺"）
- ❌ 添加译者自己的评价或注释（如需要可用脚注明确标注为"译注"）

**示例（差的翻译）：**

> 原文：The core loop is the reliability system.
>
> ❌ 核心循环本身就是那个最牛最棒最硬核的可靠性系统！
>
> ✅ 核心循环本身就是可靠性系统。

---

## 五、特殊处理

### 5.1 引用与出处

- 原文引用的人名、公司名、媒体名保留原文
- 原文提到的链接完整保留在译文的对应位置
- 参考资料/延伸阅读部分保留原始标题（英文），并为描述性文字提供翻译

### 5.2 图片说明

如原文包含图片标题或说明文字（caption），翻译说明文字，保留图片引用路径不变。

### 5.3 译注

当原文存在文化背景差异或需要额外解释时（如英文双关语、特定文化梗），可添加括号内译注：

```
（译注：此处原文利用了 "dream" 一词的双关含义）
```

译注应极少使用，仅在不注释会导致信息丧失时添加。

---

## 六、质量检查清单

翻译完成后，按以下项目自查：

- [ ] 所有专业术语首次出现时是否附注了英文原文
- [ ] 是否存在原文没有的夸张、比喻或情绪化表达
- [ ] 代码标识符、命令、路径是否保持原文未翻译
- [ ] 句式是否自然流畅，无明显翻译腔
- [ ] 标题层级、列表结构、加粗/斜体标记是否与原文一致
- [ ] 参考链接是否完整保留
- [ ] 元信息（作者、日期、原文链接）是否齐全
