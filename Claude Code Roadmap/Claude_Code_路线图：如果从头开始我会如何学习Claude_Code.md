## Claude Code 学习路径

学习路径分为五个主要的递进阶段：

- **第一阶段：基础——核心 CLI 与工作流**：掌握安装、鉴权和基本命令的基础知识。学习权限系统和 Git 集成。熟悉交互模式（interactive mode）和常见开发工作流。
- **第二阶段：配置与定制**：理解设置层级，创建能为 Claude 提供完整项目上下文的强大 CLAUDE.md 文件，构建自定义斜杠命令（slash commands），并按需配置你的终端环境。
- **第三阶段：扩展系统——释放强大能量**：这是 Claude Code 从一个实用工具蜕变为开发利器的阶段。学习子智能体（subagents）处理专项任务，通过 MCP 集成外部工具，用钩子（hooks）自动化工作流，以及利用全新的 Skills 和 Plugins 系统。
- **第四阶段：高级自动化与程序化使用**：超越交互式使用，进入全面自动化。掌握无界面模式（headless mode），在 Python 和 TypeScript 中使用 Claude Agent SDK，集成 GitHub Actions，并借助检查点和沙箱隔离等前沿功能实现更安全的自主作业。
- **第五阶段：生产与企业级部署**：在团队和生产环境中规模化部署 Claude Code。配置云服务商接入，设置企业网络，实施托管策略（managed policies），监控使用量和成本，并构建适用于整个团队的协作模式。

> **每个阶段都包含具体的子章节，涵盖实践要点、实际案例，以及将彻底改变你开发视角的关键转折时刻。**

每个阶段都建立在前一阶段的知识基础之上，是下一阶段的前提。

让我们从第一阶段开始。

## 第一阶段：基础——核心 CLI 与工作流

在尝试任何定制之前，你需要先理解 Claude Code 的运作方式。

> **跳过这一基础，你将花费数小时困惑于为什么事情没有按预期运行。**

第一阶段的重点是与核心界面建立肌肉记忆，理解权限模型，并熟练掌握基本工作流。

---

### 1.1 安装与配置

正确安装 Claude Code 可以避免后续的麻烦。

| 安装方式 | 命令 | 适用场景 |
|---|---|---|
| npm | `npm install -g @anthropic-ai/claude-code` | 有 Node.js 的开发者 |
| 原生二进制文件 | 从发布页下载 | 启动更快，无需 Node.js |
| Homebrew（Mac） | `brew install claude-code` | Mac 用户 |

---

### 1.2 CLI 命令精要

四个命令完成 90% 的工作。

- `claude`：启动交互模式——这是你与 Claude 对话的主要工作空间
- `claude -p "prompt"`：执行单条提示后退出，适用于快速任务、脚本和自动化
- `claude -c`：继续上次对话，从中断处接着往下
- `claude -r`：从检查点恢复，回滚到上一个状态

---

### 1.3 交互模式要领

交互模式是这一阶段你花费时间最多的地方。

- **Ctrl+C**：取消正在运行的操作，中途打断 Claude
- **Ctrl+D**：退出会话，干净关闭
- **Esc Esc（按两次）**：撤销/回滚改动，回退错误操作
- **Ctrl+R**：搜索提示历史，复用以前的提示

按需学习快捷键，不必一次性全部掌握。

**@提及（@mentions）**是改变工作方式的关键：

- `@filename.js`：给 Claude 提供该文件的完整上下文
- `@src/`：共享目录内的所有文件
- `@file1.js @file2.js`：从多个文件获取上下文
- 图片提及：拖放或粘贴，Claude 分析视觉内容

> **@提及功能从根本上改变了你的工作方式。输入 @ 后开始输入文件名，Claude Code 会显示匹配项。**

选择一个文件，Claude 就获得了该文件的完整上下文。你可以在一条提示中提及多个文件，这远比将代码复制粘贴到提示中来得高效。

目录提及同理。提及一个目录，Claude 就获得了其中所有文件的上下文。在处理跨多个文件的功能时非常有用。

> **图片输入功能让你可以分享截图、图表或 UI 原型。Claude 能分析这些图片并据此生成代码。丢一张设计稿让 Claude 来实现——这个工作流本身就能节省数小时。**

---

### 1.4 常见工作流

以下是你日常会用到的工作流。

**代码审查工作流：** 提及你想审查的文件，请 Claude 分析它，然后获得关于潜在问题、优化机会和最佳实践的反馈。当你沉浸在实现细节中时，Claude 能发现你可能遗漏的模式。

**功能实现模式：** 描述你想构建的内容，提及相关的现有文件作为上下文，让 Claude 生成实现代码。你来审查、测试，然后迭代。关键是提前把需求说清楚。

**bug 修复流程：** 描述 bug，分享错误信息，提及相关文件。Claude 帮你溯源问题并提出修复建议。权限系统让你可以控制 Claude 是否直接应用修复。

> **Git 操作集成是内置的。Claude 能读取你的 Git 状态、检查分支、查看 diff，甚至用规范的提交信息创建 commit。你无需切换上下文就能保持工作状态。**

---

### 1.5 权限系统

理解权限系统是这一阶段的关键转折点。

权限系统是一套工作流控制机制，让你决定 Claude 应该有多大的自主权。

> **工具审批的运作方式很简单：当 Claude 想使用某个工具（读文件、写代码、运行命令）时，如果你处于询问模式（ask mode），它会先请求许可。你能看到它计划做什么，然后批准或拒绝。**

沙箱隔离在这里初步介绍，但要到第四阶段才能充分利用。

基本概念是：你可以定义边界（哪些目录、哪些网络主机），在这些边界内 Claude 可以自由工作，无需为每个操作请求许可。

在这些边界内，Claude 运转高效；在边界之外，它会停下来询问。

> **安全最佳实践从这里开始。不要批准你不理解的 bash 命令。在允许文件写入操作之前先审查它们。从严格的权限开始，随着你在特定工作流中建立信任，再逐渐放宽。**

权限模式允许你按工具进行定制。也许你信任 Claude 读取任何文件（读操作设为 allow 模式），但想审查所有写入（写操作设为 ask 模式）。你可以根据自己的风险承受度和工作流速度需求灵活组合。

---

## 第一个关键转折点

在第一阶段，大多数人的认知突破是这样的：权限不是障碍，而是工作流控制手段。

> **刚开始时，权限感觉像是打扰。Claude 问，你批准，Claude 再问，你再批准。看起来很繁琐。**

然后你意识到，可以预先批准特定上下文中的特定操作。你在设置中配置，允许对你的源代码目录自动执行读操作。

突然间，Claude 不需要问 50 次许可就能分析整个代码库。

但你把写操作保留在询问模式下，因为你想在改动发生之前审查它们。这让你在需要速度的地方得到速度，在需要控制的地方得到控制。

> **这时 Claude Code 就从"有帮助但很慢"变成了"又快又可信"。*你找到了自主与监督之间适合你工作流的平衡点。***

> **这种理解贯穿每一个阶段。子智能体有权限，钩子有权限，MCP 服务器有权限。现在掌握这个概念，后面的一切都会自然而然地理解。**

## 第二阶段：配置与定制

基础已经打好。现在是时候让 Claude Code 按你的方式工作了。

> **第二阶段是将 Claude Code 调整为专属于你的项目、工作流和偏好的阶段。大多数开发者匆匆略过这一阶段，错过了那些每周能节省数小时的配置调整。**

这一阶段将 Claude Code 从有用变为不可或缺。

---

### 2.1 设置层级

理解设置的运作方式，可以避免浪费时间的配置冲突。

Claude Code 使用三层层级来控制设置的优先级。

- **全局设置**存储在 `~/.claude/settings.json`，适用于所有地方。这些是跟随你跨项目的基准偏好。
- **项目设置**位于仓库根目录的 `.claude/settings.json`。它们覆盖全局设置，定义 Claude Code 在这个特定代码库中的行为。大多数定制在这里完成。
- **本地设置**使用同目录下的 `.claude/settings.local.json`。这些设置覆盖全局和项目设置。用于存放你不想提交到版本控制的个人偏好。
- **优先级规则：** 本地优先于项目，项目优先于全局。当同一设置存在于多个文件时，最具体的那个生效。

环境变量也可以覆盖设置，对 CI/CD 环境或临时测试很有用，无需修改文件。

> **以下是让人踩坑的地方：如果你修改了项目设置但仍看到旧行为，检查是否有本地覆盖；或者你在编辑全局设置，其实本意是修改项目设置。这个层级至少会让每个人困惑一次。**

---

### 2.2 CLAUDE.md——项目的记忆

这个文件是项目工作中最重要的配置。

> **我喜欢将 CLAUDE.md 视为代码库的宪法。Claude 会自动读取这个文件，并将其作为了解项目运作方式的主要参考。**

该文件位于仓库根目录。Claude 在每次会话开始时加载它，提供上下文，让你无需在每条提示中重复说明。

> **CLAUDE.md 里应该写什么？项目架构概览、编码规范、依赖信息、构建和测试命令、部署流程，以及任何其他 Claude 需要了解代码库的相关内容。**

对于个人项目，可以让 Claude 在你工作的过程中逐步充实这个文件——它学习你的模式并自动记录下来。

对于专业工作，则需要有意识地维护这个文件。

> **我们 monorepo 的 CLAUDE.md 有 13KB，浓缩了我们构建软件的经验智慧。增长到 25KB 也没有问题。**

结构很重要。从高层项目概览开始，添加架构、关键组件、开发工作流和测试方法的章节，并包含项目中优质代码的示例。

差劲的 CLAUDE.md 要么过于简单（缺乏有用的上下文），要么过于冗长（让 Claude 被细节淹没）。目标是提供 Claude 高效协助你所需的信息。

> **CLAUDE.md 的关键转折时刻：当你开始新会话，Claude 立刻理解你的项目结构、命名规范和工作流，不需要任何解释。这就说明你的 CLAUDE.md 在发挥作用了。**

---

### 2.3 自定义斜杠命令

斜杠命令是频繁操作的快捷方式。

> **不必每次都键入相同的指令，你可以把它们打包成一个命令，一行调用。这既省时间，又保证一致性。**

命令以 Markdown 文件的形式存放在 `.claude/commands/` 目录下，每个文件定义一条命令，包括其行为和参数。

> **创建命令很直接。新建一个 Markdown 文件，添加包含命令名称和描述的 YAML frontmatter，然后写下调用这条命令时 Claude 应该遵循的指令。**

`$ARGUMENTS` 变量传递你在命令名后输入的内容。所以 `/review $ARGUMENTS` 让你可以写 `/review auth.js`，Claude 就知道要审查那个特定的文件。

从你最常用任务的命令开始。代码审查模板、测试工作流、文档生成或部署检查——任何你每天重复做多次的操作都是候选。

随着命令库的增长，组织至关重要。将相关命令放在子目录里，使用清晰好记的名称，并为每条命令添加说明。

> **注意事项：不要为每件事都创建命令，包括那些你很少用到的。专注于真正节省时间的场景。三条你每天都用的精良命令，胜过二十条你从来想不起来的命令。**

---

### 2.4 终端配置

小小的终端调整积累起来可以带来显著的工作流提升。

> **Shell 集成让 Claude Code 在终端中用起来像原生工具。状态栏（status line）展示 Claude 正在做什么，而不会占用你的工作空间。**

**提示历史**（Ctrl+R）经常被低估。搜索以前的提示并编辑它们，而不是重新输入。仅这一点每天就能节省好几分钟。

**状态栏**定制展示 Claude 的当前状态、token 使用量和活跃模型。把它放在不干扰日常终端工作的位置。

**主题配置**让 Claude Code 的外观与你的终端主题保持一致。深色模式用户不应在 Claude 响应时看到刺眼的白色闪光。

> **配置文件位于你的设置中。在这里做出小调整，可以防止长时间编码时的眼睛疲劳。**

---

### 2.5 模型配置

知道在什么时候用哪个模型，影响成本和效果。

Claude Code 默认使用 Sonnet 4.5，这是日常编程中最智能、最高效的模型。但你还有其他选择。

使用 `/model` 命令切换模型，根据任务复杂度和预算来选择。

- **Sonnet 4.5**：处理从常规重构到复杂架构决策的一切。将其作为默认选项。
- **Opus 4.1**：为最苛刻的任务提供最强能力。留给 Sonnet 难以应对的极复杂推理场景。
- **Haiku 3.5**：在速度比精度更重要时表现出色。快速语法修复、简单重构或重复性编辑。

> **思考 token（thinking tokens）和回退模型（fallback models）增加了另一个维度。对于需要 Claude 在响应前逐步推理的复杂问题，启用扩展思考模式。**

当主模型触达速率限制或出错时，回退模型介入。在设置中配置这一项，避免关键工作被打断。

> **成本优化的关键是为每个任务选用合适的模型。在简单任务上用 Opus 是浪费钱；在复杂架构讨论上用 Haiku 则会浪费时间在返工上。**

## 第二个关键转折点

在第二阶段，认知突破是这样的：配置不是为了好玩调调设置，而是为了消除日常工作流中的摩擦。

刚开始配置 Claude Code 时，感觉像是额外工作。为什么要花时间写 CLAUDE.md，而不是直接在提示中解释清楚？为什么要创建斜杠命令，而不是直接输入指令？

> **然后你在同一个项目上工作了一周。你意识到自己每次会话都在重复解释同样的架构，为每个文件输入类似的审查指令，还在手动根据任务复杂度切换模型。**

那时配置就有了回报。

这一基础为第三阶段做好了准备——在那里，Claude Code 通过子智能体、MCP 集成、钩子以及全新的 Skills 和 Plugins 系统释放出真正的威力。

> **那才是真正有趣的地方。**

## 第三阶段：扩展系统

第三阶段引入了数倍放大 Claude Code 能力的扩展系统。

> **子智能体处理专项任务。MCP 连接外部工具和数据。钩子在关键生命周期节点自动化工作流。Skills 打包可复用的专业能力。Plugins 将一切捆绑在一起。**

大多数开发者永远无法到达这一阶段——他们停留在基础交互模式，错失了每天能节省数小时的自动化潜力。

你即将解锁让 Claude Code 与众不同的东西。

### 3.1 子智能体（Subagents）

子智能体是处理专项任务的专业 AI 助手。

> **我喜欢把子智能体想象成我的专家团队。与其让 Claude 这一个通才尝试做所有事情，不如把任务委派给为特定工作优化的专家。**

每个子智能体在自己的上下文窗口（context window）中运行。这防止了主对话中的上下文污染，让交互保持对高层目标的专注。

> **创建子智能体需要在 `.claude/agents/` 目录下将其定义为带有 YAML frontmatter 的 Markdown 文件。frontmatter 指定名称、描述以及可选的工具访问权限。**

Markdown 内部的系统提示定义了子智能体的行为方式。要具体。代码审查子智能体需要的指令不同于测试子智能体或安全审计子智能体。

你可以只给子智能体访问特定工具的权限。文档子智能体可能只需要读权限，而不需要写权限；部署子智能体需要写权限，但只能访问特定目录。

> **当 Claude 识别出与子智能体描述匹配的任务时，它会自动将任务委派给相应的子智能体。你也可以显式调用它们："使用 code-review 子智能体检查 auth.js"。**

对于复杂项目，上下文隔离的价值尤为明显。你的主智能体保持大局观，而子智能体则深入专项工作并返回摘要。

**常见的子智能体模式：** 代码审查专家、测试编写者、文档生成器、安全审计员、重构专家和架构顾问。每一个都针对其特定领域进行了调校。

> ***注意事项：不要过早创建太多子智能体。先从一到两个用于你最频繁的专项任务的开始，随着模式浮现再逐步增加。***

### 3.2 模型上下文协议（Model Context Protocol，MCP）

MCP 通过标准化协议将 Claude Code 连接到外部工具和数据源。

> **没有 MCP，Claude 只知道你本地文件里的内容。有了 MCP，Claude 能查询 GitHub issues、搜索文档、访问数据库，并与任何提供 MCP 服务器的服务交互。**

使用 `/mcp` 命令安装 MCP 服务器。热门服务器包括：用于仓库操作的 GitHub、用于获取最新库文档的 Context7，以及用于数据库访问的 Supabase。

> **每个 MCP 服务器暴露 Claude 可以使用的工具。GitHub MCP 服务器提供搜索 issues、创建 PR 和读取仓库数据的工具；数据库 MCP 服务器提供查询执行和 schema 检查功能。**

2025 年中期，远程 MCP 支持到来，你不再需要本地服务器配置——直接指向托管的 MCP 服务器即可开始使用。OAuth 认证处理安全连接，MCP 服务器单独管理凭据，保持与 Claude Code 之间的安全边界。

> **威力来自于组合多个 MCP 服务器。Claude 可以通过 Context7 搜索最新的库更新，在你的 GitHub 仓库中检查实现模式，并从你的数据库查询生产数据——全都在同一个工作流中完成。**

**安全注意事项：** MCP 服务器对其连接的服务拥有完整访问权限，只安装你信任的服务器，查看它们暴露的工具，了解它们可以访问哪些数据。

**热门的开发 MCP 服务器：** 用于仓库操作的 GitHub、用于库文档的 Context7、用于高级文件操作的 filesystem，以及用于公司特定工具的自定义内部服务器。

> **当你意识到 Claude 现在能访问你的整个开发生态系统——而不只是本地文件——那些看似不可能的工作流就变得简单了。**

### 3.3 钩子系统（Hooks System）

钩子让你在 Claude Code 生命周期的关键节点运行代码。

> **钩子本质上是自动化触发器。当特定事件发生时，你的自定义脚本自动执行。这使得能够适应团队流程的工作流成为可能。**

钩子配置存储在你的设置文件中。你定义哪些事件触发哪些钩子，以及这些钩子应该做什么。

> **可用的事件包括：PreToolUse（Claude 使用工具前）、PostToolUse（工具完成后）、Stop（Claude 完成时）、SubagentStop（子智能体完成时），以及其他几个事件。**

存在两种钩子类型：运行 bash 命令的命令钩子（command hooks），以及使用大语言模型进行智能决策的提示钩子（prompt hooks）。

命令钩子执行 shell 脚本，用于确定性操作，如验证、日志记录或触发外部系统。

> **提示钩子将当前情况发送给 LLM 进行评估。LLM 决定批准、阻止还是修改该操作，从而实现上下文感知的工作流控制。**

**钩子的实际应用：** 在提交前强制代码审查、在部署前验证测试已运行、为审计追踪记录所有文件修改、阻止对受保护文件的变更，或根据任务注入额外上下文。

MCP 工具模式与钩子配合使用：你可以挂载 `mcp__github__create_pr` 在 PR 创建前运行验证，或挂载 `mcp__database__execute_query` 记录所有数据库操作。

> **钩子参考文档涵盖所有事件名称及其对应的触发时机，在构建复杂钩子工作流之前，请仔细研读这份文档。**

### 3.4 Skills 系统

Skills 是打包了指令、脚本和资源以提供特定能力的文件夹。

> **Skills 通过专业知识扩展 Claude 的能力。当 Claude 遇到匹配某个 skill 的任务时，它会自动按需加载。**

***Skills 以 Markdown 文件（SKILL.md）作为入口点，该文件包含 skill 所需的指令、示例以及对相关脚本或资源的引用。***

如果你使用过 Claude 应用，那么你已经见过 Skills 在运行了——文件创建功能（电子表格、演示文稿、文档）就是基于 Skills 运行的。

**安装 Skills** 通过插件系统完成，或者手动将其放置在 `~/.claude/skills` 目录下。Claude 会扫描可用的 skills，并在需要时加载相关的那些。

**可组合性**是关键优势。Skills 会自动叠加。Claude 识别哪些 skills 适用于你当前的任务，并在无需明确指示的情况下协调使用它们。

**创建自定义 Skills** 让你可以打包团队的专业知识——品牌指南、代码审查清单、部署流程或安全审计协议都可以成为 Skills。

**可移植性**意味着 Skills 在任何地方都能用。只需构建一次，即可跨 Claude 应用、Claude Code 和 API 集成使用，格式通用。

效率来自按需加载。Claude 只在需要时加载所需内容，在访问专项能力的同时保持性能。

> **安全提示：Skills 可以执行代码，只使用可信来源的，或者在安装前仔细审查 Skills 的内容。**

### 3.5 Plugins

Plugins 将斜杠命令、子智能体、MCP 服务器、钩子和 Skills 捆绑成可分发的包。

> **Plugins 解决的是共享问题。你构建了一套结合多种扩展类型的强大工作流，然后将其打包成一个 plugin，别人一步就能安装。**

插件安装在公测阶段使用 `/plugin` 命令。浏览可用插件，安装你需要的，然后根据需求随时开启或关闭。

> **一个 plugin 里可以包含什么？扩展系统的任意组合。一个部署 plugin 可能包含：常见操作的斜杠命令、用于复杂部署的子智能体、用于安全检查的钩子，以及与你云服务商 MCP 的集成。**

使用场景跨越团队和社区。工程领导者在团队中分发标准化工作流；开源维护者提供帮助开发者正确使用其包的 plugins；框架作者打包使用其框架所需的一切。

Plugin 市场让发现变得容易。任何人都可以构建和托管 plugins，团队可以创建内部市场来管理公司专有工具。

> **开关功能很有帮助：在需要其能力时启用 plugin，不需要时禁用，以减少系统提示的复杂性。**

### 3.6 综合运用

扩展系统在组合使用时威力倍增。

> **一个生产工作流可能是这样的：你使用一个专门进行代码审查的子智能体，通过 MCP 连接 GitHub 获取 issue 上下文，以及连接文档系统获取规范标准。钩子确保审查遵循团队清单。Skills 提供编码规范知识。所有这些都被打包成一个 plugin，新开发者一步安装即得。**

四个扩展系统协同运作。结果是：以前需要 30 分钟的全面代码审查，现在只需 3 分钟，一致性还更好。

### 4.1 无界面模式（Headless Mode）

无界面模式让你无需交互提示就能运行 Claude Code，非常适合脚本和自动化。

> **`--output-format json` 标志将 Claude Code 从一个对话界面转变为一个返回可解析、可处理结构化数据的可编程工具。**

使用场景涵盖各种自动化工作流。

> **从代码变更生成文档；分析 pull request 查找问题；按计划运行安全审计；以一致的操作批量处理文件。**

JSON 输出包含 Claude 以结构化格式返回的所有信息，可以用 `jq` 等标准工具解析，或在你选择的语言中进行原生 JSON 解析。

> **退出码：Claude Code 成功时返回 0，出错时返回非零值，可与 shell 脚本和 CI/CD 流水线集成。**

即使在无界面模式下，Claude Code 也可以流式输出响应，让你能在生成时逐步处理，而不必等到完成。

错误处理在无界面模式下至关重要。用 try-catch 块包裹调用或检查退出码，记录失败，对瞬时错误实现重试。

> **实用自动化模式：一个定时任务运行无界面 Claude Code 分析近期 commits，生成摘要并发布到 Slack。无需人工干预。**

---

### 4.2 Claude Agent SDK——Python

Python SDK 让你能以编程方式控制 Claude Code 的全部能力。

> **可以把 SDK 看作暴露为 Python 库的 Claude Code。所有你能交互式完成的事情，现在都可以用 Python 代码脚本化和自动化。**

安装很简单。安装包，导入客户端，开始构建。

> **`ClaudeSDKClient` 提供主要接口。用你的配置初始化它，然后用 `ClaudeCodeOperations` 以编程方式执行任务。**

消息类型支持不同的交互模式：发送单条提示、维护对话，或以异步方式流式处理响应以进行实时处理。

> **异步流式处理用于处理长时间运行的操作。你的代码可以在 Claude 生成响应时进行处理，更新进度指示器或对中间结果做出反应。**

**实际自动化示例：** 一个 Python 脚本，审查目录中的所有文件，应用一致的格式化，并生成汇总报告；一个分析 API 文档并生成客户端库代码的工具；一个监控错误日志并自动提出修复建议的系统。

SDK 包含 Claude Code 的所有功能。子智能体、MCP 服务器、钩子和设置都能以编程方式工作。构建工具来编排结合多种扩展系统的复杂工作流。

> **常见模式：一个 Python 服务监听 GitHub webhooks，使用 Claude Code SDK 分析变更，并创建带有建议的审查评论。全天候运行，无需人工干预。**

---

### 4.3 Claude Agent SDK——TypeScript

TypeScript SDK 将同样的程序化能力带到了 JavaScript 生态系统。

> **Node.js 开发者通过一套良好类型化的 API 获得对 Claude Code 的第一类访问，与 JavaScript 工具链自然集成。**

配置与 Python 版本类似，安装包、导入必要组件，然后开始构建自动化工作流。

查询函数提供核心接口，发送提示、管理对话，并借助 TypeScript 出色的类型安全处理响应，确保不遗漏边界情况。

消息处理支持多种模式：单次请求、持久对话，以及用于实时 UI 更新的流式响应。

> **SDK MCP 服务器集成意味着你的 TypeScript 工具可以利用 MCP 生态系统——像交互式 Claude Code 一样连接数据源、外部 API 和服务。**

为 Web 应用、命令行工具或后端服务构建工具，TypeScript SDK 兼容所有 Node.js 运行的地方。

> **实际应用：一个 Next.js 应用，让用户用自然语言描述功能，使用 Claude Code SDK 生成组件代码，并展示实时预览。整个流程在你的应用内运行，无需交互式启动 Claude Code。**

### 4.4 GitHub Actions 集成

GitHub Actions 中的 Claude Code 在仓库层面自动化代码审查、测试和维护。

> **GitHub Actions 集成在你的 CI/CD 流水线中运行 Claude Code，为每次 pull request、commit 或定时事件提供自动化辅助。**

配置需要在 `.github/workflows/` 下创建工作流文件，定义何时运行该 action、它应该做什么，以及如何上报结果。

该 action 在一个检出了你仓库的干净容器中运行。Claude Code 需要访问你的代码以及 MCP 服务器或外部服务所需的 secrets。

> **PR 审查自动化是杀手级用例。每个 pull request 都会触发 Claude Code 审查变更、检查编码规范、识别潜在问题并发布审查评论——这一切都在人工审查者看到代码之前就完成了。**

Issue 分流模式帮助管理大型项目——Claude Code 可以分析新 issue、将其分类、建议相关标签，甚至为常见问题提出初步修复方案。

工作流配置控制行为。可以在每次推送时运行，只针对主分支的 PR，或按计划执行。自定义 Claude Code 检查的内容和上报结果的方式。

> **沙箱隔离和安全在 CI/CD 环境中变得至关重要。GitHub Actions 容器提供隔离，但要确保你的工作流不暴露 secrets 或赋予过多权限。**

高级模式组合多个 actions：Claude Code 审查代码，另一个 action 运行测试，第三个检查文档完整性，最后一个汇总所有反馈生成综合报告。

> **目标是在人工审查者介入之前，pull request 就已经附带自动审查反馈——节省数小时并及早发现问题。**

### 4.5 检查点（Checkpoints）

检查点自动保存你的代码状态，让你能自信地探索改动，在需要时即刻回滚。

> **检查点解决了"万一 Claude 把一切都搞坏了怎么办"的问题，这个问题曾经限制了自主作业的边界。**

在 Claude 进行每次改动之前，系统会自动保存状态，让你可以无所顾虑地探索大胆的重构。

> **回滚功能有两种方式：按两次 Esc，或使用 `/rewind` 命令，两者都能立即带你回到上一个检查点。**

回滚时，你可以选择恢复什么。只回滚代码但保留对话上下文；同时恢复代码和对话以便以不同方式重演；或只恢复对话以从不同分支点继续。

**重要区别：** *检查点追踪 Claude 的编辑，而不是你的手动编辑或 bash 命令。将它们与 Git 配合使用，而不是作为替代。*

> **最佳实践：在重大变更前设置检查点，结合 Git 保留完整历史，当探索出错时使用回滚，而不是试图手动撤销改动。**

---

### 4.6 沙箱隔离（Sandboxing）

沙箱隔离创建了安全边界，让 Claude 在其中自由工作，无需不断请求许可。

> **沙箱隔离通过利用隔离来解决权限疲劳（permission fatigue）问题。**

`/sandbox` 命令用于配置沙箱环境，定义 Claude 可以自主运行的文件系统边界（哪些目录）和网络边界（哪些主机）。

文件系统隔离确保 Claude 只接触特定目录，在这些边界之外，它必须请求许可；在边界之内，它可以自由工作。

网络隔离控制外部连接。Claude 可以自动连接已批准的服务器，但会阻止未批准的目标。

> **安全模型防止提示注入攻击。即使攻击者通过注入提示攻陷了 Claude，沙箱也会阻止其访问敏感文件或网络资源。**

许可提示减少了 84%，让 Claude Code 在实际感受上明显更快。以前需要不断审批的操作，在沙箱边界内运行时现在流畅进行。

> **配置策略：从窄范围的沙箱边界开始，随着建立信任逐步扩大。初始时过于严格比过于宽松要好。**

---

### 4.7 自主功能（Autonomous Features）

近期更新让 Claude Code 能够更自主地处理复杂的长时间运行任务。

> **并行工具执行让 Claude 能同时使用多个工具。Claude 可以同时读取多个文件，而不是依次读取；可以并行执行多个 bash 命令，而不是一次运行一个。**

多 bash 命令执行专为开发工作流设计——在构建文档的同时运行测试，在运行类型检查的同时执行 lint。在大型代码库上，节省的时间会累积放大。

> **扩展的自主运行能力意味着 Claude Code 可以处理跨越多个步骤的复杂任务，无需持续的人工干预。检查点（安全回滚）、沙箱隔离（安全边界）和并行执行（速度）的组合，使真正的自主作业成为可能。**

监控自主作业变得至关重要。关注 Claude 的进展，检查中间结果，如果方向出现偏差就介入。目标是受监督的自主，而不是盲目信任。

> **配置控制自主级别。对敏感代码库使用严格边界，对实验性项目使用宽松权限，根据风险承受度和任务复杂性进行调整。**

在第四阶段，认知突破是这样的：Claude Code 不只是你使用的工具，它是持续运行的基础设施。

进入第四阶段时，自动化看起来有些过头。既然可以直接跟它对话，为什么还要用脚本操作 Claude Code 呢？

> **然后你构建了第一个真实的自动化。也许是一个审查每个 PR 的 GitHub Action，也许是一个维护文档的 Python 脚本，也许是一个按计划分析日志并提出优化建议的定时任务。**

然后你意识到每次运行都节省了本来要花在重复工作上的时间。

那时第四阶段的价值就清晰了。

那就是 Claude Code 成为企业基础设施的地方。

## 第五阶段：生产与企业级部署

你已经掌握了个人使用的 Claude Code，现在是时候在团队和生产环境中规模化部署了。

> **第五阶段是 Claude Code 从个人开发工具升级为组织基础设施的地方。云服务商集成、企业网络兼容性、托管策略、监控与成本追踪，以及规模化运作的团队协作模式。**

大多数组织从未到达这一阶段。那些到达第五阶段的，在整个工程团队中解锁了复合收益。

这是对 Claude Code 投入带来指数级回报的地方。

---

### 5.1 云服务商集成

企业级部署通常需要使用云托管模型，而不是直接访问 API。

> Amazon Bedrock 和 Google Cloud Vertex AI 让你能在现有的云基础设施内运行 Claude 模型，集成你组织的合规、安全和计费系统。

**Amazon Bedrock** 配置将 Claude Code 连接到你的 AWS 账户——配置区域，设置 IAM 权限，让 Claude Code 通过 Bedrock 路由请求，而不是直接访问 Anthropic 的 API。

> **Google Cloud Vertex AI 遵循类似的模式。用你的 GCP 账户进行身份验证，选择区域，然后配置 Claude Code 使用 Vertex AI 作为模型提供商。**

区域选择影响延迟和数据驻留地。选择靠近你团队的区域以获得更好的响应时间，并考虑受监管行业的数据主权要求。

凭证管理与云服务商工具集成——使用 IAM 角色、服务账户或 Workload Identity，而不是硬编码凭证，这样既维护了安全性，又支持凭证轮换。

集成云服务商后，成本分配变得更容易。Bedrock 和 Vertex 的使用量出现在你现有的云账单中，让你能够与其他基础设施费用一起追踪 Claude Code 的成本。

> **企业优势：合规团队熟悉云安全控制；财务团队获得统一账单；运维团队获得一致的监控；工程团队获得同样的 Claude Code 能力。**

---

### 5.2 VS Code 扩展

原生 VS Code 扩展将 Claude Code 直接带入你的 IDE，并提供图形化功能。

> **该扩展为偏好 IDE 工作流而非基于终端交互的开发者，提供了更全面的界面。**

通过 VS Code 扩展市场安装——搜索 Claude Code，安装，完成身份验证，扩展就会出现在你的侧边栏中。

> **侧边栏面板实时显示 Claude 的工作状态。观察 Claude 编辑文件、运行命令、进行变更。视觉反馈帮助你追踪复杂操作的进展。**

内联 diff 直接在编辑器中显示变更，无需切换上下文即可看到 Claude 修改了什么，用标准的 VS Code diff 控件接受或拒绝变更。

> **扩展与终端工作流的选择取决于个人偏好。扩展提供视觉清晰度和编辑器集成；终端提供速度和可脚本化。许多开发者根据任务两者兼用。**

---

### 5.3 企业网络配置

企业网络在与外部 AI 服务配合使用时需要特殊配置。

> **代理设置、防火墙规则、SSL 证书处理，以及 LLM 网关集成，在企业环境中部署 Claude Code 时都会变得相关。**

**代理配置**告诉 Claude Code 如何通过企业代理路由流量，设置代理 URL、所需的身份验证以及任何 SSL/TLS 设置。

**SSL 证书**处理在你的组织使用自定义证书颁发机构时至关重要——配置 Claude Code 信任你的内部证书以进行 HTTPS 连接。

**LLM 网关**将 Claude Code 通过你组织的 AI 网关进行路由，这个集中控制点支持对所有 AI 服务使用进行监控、日志记录和策略执行。

**防火墙注意事项**需要开放特定端点，与网络团队合作将必要的域名加入白名单，并为安全审查清晰记录这些要求。

> **配置通常发生在全局设置层面。所有开发者安装 Claude Code 时会自动继承企业网络设置。**

---

### 5.4 托管策略与治理（Managed Policies & Governance）

组织需要在所有 Claude Code 安装实例中执行一致的策略。

> **托管策略（managed policies）强制执行开发者无法覆盖的规则，确保合规性、安全标准和成本控制普遍适用。**

策略执行利用设置层级。部署覆盖个人偏好的组织级设置，开发者可以在限定范围内定制，但无法禁用必需的策略。

> **团队范围的设置分发通过版本控制完成，将设置存储在中央仓库中，团队克隆并引用这些设置。**

审计日志捕获 Claude Code 使用情况以满足合规要求——追踪运行了哪些命令、哪些文件发生了变更、访问了哪些外部服务，并将日志与你的监控系统集成。

> **合规要求驱动许多策略：阻止 Claude Code 访问敏感目录、要求生产部署前进行审批、屏蔽特定 MCP 服务器、记录所有数据库操作。**

策略设计需要在控制与生产力之间取得平衡。过于严格，开发者就会绕开工具；过于宽松，你就失去了治理。通过迭代优化找到中间点。

***定期策略审查让配置保持与时俱进。适合 10 个开发者的方案可能无法扩展到 100 人。随着使用模式的出现，调整策略。***

---

### 5.5 监控与分析（Monitoring & Analytics）

了解 Claude Code 的使用模式、成本和效果，需要监控与分析。

> **Claude Code Analytics API 提供每日使用追踪、成本分析、性能指标和 token 消耗模式。**

每日使用追踪揭示哪些团队最多使用 Claude Code、他们利用哪些功能，以及使用趋势如何演变——这些数据为培训优先级和资源分配提供参考。

> **成本分析按团队、项目和模型细分支出。识别高成本操作，优化模型选择，准确预算 Claude Code 基础设施费用。**

性能指标揭示瓶颈——追踪响应时间、成功率和错误频率以优化性能，发现失败模式，基于数据改进工作流。

> **token 消耗模式揭示 token 的用途。识别消耗过多上下文的低效提示，通过优化 CLAUDE.md 文件或改进子智能体设计，找到可以精简的地方。**

对于需要分摊 AI 成本的团队，客户报告很有价值——生成展示 Claude Code 交付价值的报告，向干系人呈现包括完成代码审查数量、发现 bug 数量和节省开发时间在内的指标。

> **监控集成将分析数据连接到你现有的可观测性栈。将指标导出到 Datadog、Grafana 或你偏好的平台，对异常使用模式或成本飙升设置告警。**

### 5.6 团队协作模式

在团队中规模化推广 Claude Code，需要建立行之有效的协作模式。

> **配置共享、入职流程、最佳实践文档和知识传递，在组织规模下都变得至关重要。**

> **配置版本控制将 `.claude/` 目录纳入你的仓库，团队通过 Git 共享设置、子智能体、钩子和命令。新开发者自动获得团队标准。**

Plugin 分发将团队特定的工作流作为可安装的 plugins 交付，打包你团队积累的 Claude Code 专业经验。*新成员安装团队 plugin 后，立即获得对已建立模式的访问权限。*

入职新团队成员时，将 Claude Code 作为第一等工具对待——文档、培训和辅导确保每个人都能快速达到至少第二阶段的熟练程度。

> **最佳实践文档捕获有效的方法：维护常见工作流的内部指南，分享解决特定问题的子智能体模式，记录何时使用哪些功能。**

知识传递通过展示分享自然发生：开发者演示他们精妙的 Claude Code 工作流，团队互相学习，专业知识有机地扩散开来。

围绕 Claude Code 建立社群，通过内部 Slack 频道、定期开放答疑时间以及随时解答问题的专属负责人来维持势头。*该工具就此成为团队文化不可或缺的组成部分。*

> **协作的乘数效应：当团队共享关于 Claude Code 的知识时，效能的提升比个人学习曲线快得多。**
