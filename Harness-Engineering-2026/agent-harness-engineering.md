# 信噪比、分层与 Harness：一份关于 Agent 上下文工程的思考笔记

> 本文整理自一次围绕 Agent 上下文工程的深度讨论，涉及信噪比管理、Claude Code 的分层实现机制、Harness 工程的演化趋势，以及跨模型 Agent 的设计原则。

## 引子：一个反直觉的起点

讨论从一个常见的疑问开始：**在一次与模型的对话中，上下文长度占用是不是越少，模型的约束越好？**

这是一个看起来合理、实际上有问题的假设。答案需要拆成两个维度来看：

- **绝对长度**：确实存在上限，接近窗口极限（比如 200k）时，模型在信息检索上会衰减，出现 "lost in the middle" 效应。
- **内容质量**：同样 10k tokens，一份结构清晰、指令明确、例子精准的 prompt，远比 50k tokens 的冗余材料约束力更强。

**更准确的说法是**：在保证必要信息完整的前提下，越精炼越好。纯粹追求短反而可能削弱约束——因为你会被迫删掉角色定义、关键示例、约束条件、格式规范等真正提供约束力的内容。模型信息不足时会"猜"，自由度反而更大。

**核心结论**：约束力来自指令的明确性、示例的代表性、结构的清晰性，而不是篇幅长短。这个认知校正是后续所有讨论的基础。

## 一、Agent 场景下的信噪比管理

### 问题的转移

当上下文从"静态 prompt 设计"变成"Agent 运行时动态拼装"，信噪比的控制权也随之转移——从 prompt 设计者手里，交到了**运行时的上下文管理策略**手里。

这是质的变化。Prompt 工程可以反复打磨一个字符串，而 Agent 工程必须设计**一套在运行过程中自动维护信噪比的机制**。

### 六个关键设计维度

**1. 上下文的分层与生命周期管理**

不要把所有信息平铺，而是按存活周期分层：

- **永久层**：系统指令、角色定义、核心约束——会话全程存在
- **任务层**：当前任务目标、计划、关键中间结论——任务周期内存在
- **工作层**：工具调用的原始返回、临时数据——用完即丢或压缩归档
- **归档层**：已完成子任务的摘要，而非原始内容

关键点：工作层的原始数据不应无限累积到任务层。5000 行文件读取后，留下的应是"我从 X 文件提取到了 Y 信息"，而不是那 5000 行。

**2. 主动的上下文压缩**

- 滑动窗口 + 摘要
- 工具结果折叠
- 检查点机制生成状态快照

**3. 检索式注入，而非预先堆砌**

文件系统、代码库作为外部记忆，Agent 通过工具按需拉取。粒度要细——读特定函数而不是整个文件。

**4. 显式的规划与反思**

- 计划先行，产出 TODO list
- 状态外化到文件或 scratchpad
- 阶段性反思替代累积历史

**5. 子 Agent 隔离**

主 Agent 保持干净，子 Agent 承担脏活，完成后**只返回结论**。子 Agent 内部可能消耗数万 token 探索，主线只留一段摘要。

**6. 工具设计本身就是信噪比管理**

这点经常被忽略：**工具返回值的设计直接决定使用它的 Agent 的上下文信噪比**。

- 工具返回应结构化、摘要优先、可分页
- 避免一次性 dump 大量原始数据
- 提供元信息让 Agent 决定是否需要更多

一个设计糟糕的工具会污染所有使用它的 Agent；一个设计好的工具（支持过滤、分页、字段选择）让 Agent 天然保持干净。

## 二、什么是一份好的 Prompt

抽象原则不够具体，下面是一份可实际运行的例子，以"代码审查助手"为场景。

### 示例 Prompt

~~~markdown
# 角色

你是一位资深的代码审查者，专注于发现代码中的实际问题，而非风格偏好。
你的审查对象是团队成员提交的 Pull Request 改动。

# 任务

分析用户提供的代码 diff，识别其中的问题，并按严重程度分类输出。

# 审查维度（按优先级排序）

1. **正确性**：逻辑错误、边界条件遗漏、空指针、竞态条件
2. **安全性**：注入风险、权限绕过、敏感信息泄露
3. **性能**：明显的 N+1 查询、不必要的循环嵌套、内存泄漏
4. **可维护性**：命名歧义、过深嵌套、职责不清

**不要**评论以下内容：
- 代码风格（缩进、空格、引号类型）——由 formatter 处理
- 个人偏好性的重构建议
- 已有代码的历史问题（只看本次 diff）

# 输出格式

严格按以下 JSON 格式返回：

{
  "summary": "一句话总结",
  "issues": [{
    "severity": "critical | major | minor",
    "category": "correctness | security | performance | maintainability",
    "file": "文件路径", "line": 行号,
    "problem": "问题描述", "suggestion": "修复建议"
  }],
  "approval": "approve | request_changes | comment"
}

审批规则：
- 存在任何 critical → request_changes
- 存在 major 但无 critical → comment
- 只有 minor 或无问题 → approve

# 示例

[包含 SQL 注入 + N+1 查询的典型 diff，及期望的 JSON 输出]

# 边界情况

- diff 为空：返回空 issues，approve
- 只改注释文档：同上
- 缺少上下文：severity 设为 minor，problem 中说明
~~~

### 为什么这是一份好的 Prompt

**角色定义是约束性的，而非装饰性的**

很多 prompt 的角色写成"你是一个专业的 XX 专家"，这种话对模型行为几乎没有影响。好的角色描述要能直接映射到行为决策——"专注于发现实际问题，而非风格偏好"这一句，直接框定了"哪些该做、哪些不该做"。

**任务分解有明确的优先级和负面清单**

负面清单（negative constraints）是很多人忽略的部分。只说"评审代码质量"，模型会倾向于评论所有它能评论的东西。**显式禁止比隐式期望可靠得多**。

**输出格式是机器可解析的**

- JSON 而非自然语言——下游可直接消费
- 枚举值明确——避免模型自创
- 规则可推导——不需要模型"判断"，只需"执行"

**Few-shot 示例是"代表性"的**

好的示例要覆盖核心决策点、展示期望的详细程度、包含容易出错的边界。选一个同时包含安全和性能问题的 diff，比选十个简单的命名问题更有价值。

**边界情况被显式处理**

没有这部分，模型遇到空 diff 可能会硬编出问题来——这正是幻觉的温床。提前列出"奇怪输入"的处理方式。

**结构本身帮助模型分阶段思考**

角色 → 任务 → 维度与优先级 → 输出格式 → 示例 → 边界。这个顺序匹配模型的推理流程，不是匹配你的思考流程。

## 三、Claude Code 的真实实现：一个参考案例

讨论中一个重要的纠正：上面提到的"永久层/任务层/工作层/归档层"是**通用 Agent 设计模型**，不是 Claude Code 官方术语。Claude Code 公开文档使用的词汇是 **compaction（压缩）**、**tool-result clearing（工具结果清理）**、**memory（持久化记忆）**。

下面把两套模型对照起来，用 Claude Code 的真实机制回答"上下文分层生命周期管理"的问题。

### 会话启动：两个持久化来源

每次 Claude Code 会话都以全新的上下文窗口开始，两个机制跨会话带入知识：

- **CLAUDE.md**：用户写的持久化指令（项目结构、编码规范、禁止事项）
- **Auto memory**：Claude 根据用户纠正自动记录的偏好

关键细节：**项目根的 CLAUDE.md 在 compaction 后会从磁盘重新读取并重新注入**。它之所以"永久"，不是因为常驻上下文，而是因为每次压缩后都会被重新注入。

### 会话进行中：分层的消息队列

| 消息类型 | 真实处理方式 | 对应层 |
|---------|-------------|-------|
| System prompt、CLAUDE.md | 每次重新注入，不参与压缩 | 永久层 |
| 用户消息、助手的规划/决策 | 尽量保留，压缩时总结 | 任务层 |
| Tool call 和 tool_result 块 | 标记为可压缩，优先清理 | 工作层 |
| TodoWrite 状态 | 外化到专门的数据结构 | 任务层（外化） |

### Tool-Result Clearing：工作层的显式清理

这是 Claude Code 一个很具体的机制。官方说法是：工具结果清理处理工具使用本身带来的膨胀；清理丢弃旧的、可重新获取的结果，同时保留"调用发生过"的记录。

三个关键点：

- 丢弃的是**结果**（result），不是**调用记录**
- 判断标准是"**可重新获取**"——需要时可再调用一次工具
- 保留"调用发生过"意味着模型知道读过哪个文件，只是不再占字节

### Compaction：任务层到归档层的过渡

和 tool-result clearing 不是一回事：context editing 在客户端清理特定工具结果，compaction 在服务端自动总结整个对话。

Anthropic 给出的压缩保留优先级：

1. **用户意图** — 最高
2. **已完成的工作**
3. **错误**（避免重复踩坑）
4. **活跃的工作**
5. **关键引用**（关键文件、函数、决策）

### Subagent：隔离的工作上下文

Subagent 不是命名的 prompt 模板，而是**隔离的工作上下文**——有自己的上下文窗口、prompt、工具权限。子 Agent 完成后**只有结论回到主上下文**，内部的 tool calls、思考过程不会污染主 Agent。

### Memory Tool：跨会话归档

Memory 是结构化笔记：Agent 写入持久化外部存储，跨任务和会话跟踪进度。但要注意：memory tool 存在，但不是默认自动触发的。真正的"归档层"需要**显式写入**（通过 CLAUDE.md、外部文件或 memory tool）。

### 一个运行时的完整故事

假设用 Claude Code 重构大型模块，跑了两小时：

1. **启动**：读取 CLAUDE.md、Auto memory（永久层加载）
2. **规划**：TodoWrite 外化任务列表
3. **工作层膨胀**：Read 了 5 个文件（1 万行）、Bash 测试日志、Grep 匹配结果
4. **Tool-result clearing 触发**：最早的 Read 结果被清理，保留"读过这些文件"
5. **Compaction 触发**（约 80% 窗口）：用户意图、已完成工作、错误、活跃工作被压缩成总结
6. **压缩后**：CLAUDE.md 重新注入，TodoWrite 状态完整保留

## 四、Harness 工程的演化：一个重要的认知校正

讨论中的一个关键观察被验证：**模型确实在"吸收"一些原本属于 harness 的能力**，但这件事需要精确化——不是零和博弈，而是**三层同时在动**。

### 模型正在内置的能力

- **Context awareness**：Claude Sonnet 4.5+ 能跟踪剩余 token 预算，原本属于 harness 职责
- **原生工具使用**：模型被专门训练理解工具调用循环，不再需要复杂 prompt 教它
- **长任务持续性**：模型被训练"持续到最后"，减少 harness 督促工作

### API 平台也在吸收 harness 能力

这是一个容易被忽略的反向趋势。Anthropic 把一些 harness 能力打包成了 API 特性：

- **Context editing / tool-result clearing** — API 自动做
- **Server-side compaction** — API 自动做
- **Memory tool** — 标准化接口
- **Tool search tool** — 避免工具定义爆炸
- **Prompt caching** — 自动优化

**Harness 没有消失，而是从应用层下沉到了平台层。**

### 但 Claude Code 的 Harness 反而在变复杂

如果 harness 真的在被"吃掉"，Claude Code 应该变简单，但事实相反：

- MCP 工具定义延迟加载、按需通过 tool search 发现
- Skills 按需加载，会话开始只看描述
- Subagent 拥有独立上下文
- Hooks、权限模式、settings scope、checkpoints

这些都是近期**新加**的 harness 机制。

### 最终，Harness 工程会剩下什么

**会被模型吸收**：循环控制、基础预算管理、工具调用语法、基础错误重试、思维链结构

**会被平台吸收**：context editing、compaction、memory 接口、tool search、prompt 缓存

**真正属于 Harness 工程核心**：

1. **领域知识的注入方式** — 这是你的 Agent 和别人的 Agent 的根本区别
2. **工具的设计** — 外部世界和模型之间的接口
3. **权限与安全边界** — 谁能调什么、什么需要人类确认、什么禁止
4. **业务流程编排** — 何时用 subagent、何时走主流程、Skill 间协作
5. **可观测性和调试** — 日志、trace、审计
6. **外部系统集成** — MCP 连接、数据源、凭证管理
7. **评估和质量保证** — 怎么知道 Agent 真的做对了

**Harness 工程会从"补齐模型能力"转向"塑造 Agent 身份"。** 早期是"让裸模型变得能用"，未来是"让能用的模型变得**这个样子**"。

## 五、跨模型兼容：为什么 Harness 反而要更精心

结论明确：要兼容多个模型，harness **要比以前更精心**，而不是更简单。三个原因：

**1. 模型内置能力在分化而非收敛**

不同厂商内置不同的东西。如果 harness 假设了某个模型的内置行为，切模型时会出问题。典型风险：依赖 Claude 的"持续到最后"训练特性，换个不那么坚持的模型，Agent 会提前放弃。

**2. API 层能力不对称**

Anthropic 有 context editing、memory tool、tool search；其他厂商可能没有或语义不同。直接用了 server-side compaction，换模型时要么重写，要么自己实现等价物。

**3. 工具调用的语义差异**

表面上大家都支持 function calling，但在**稳定性、并行调用支持、错误恢复行为**上差异很大。

### 跨模型 Harness 的六条设计原则

**1. 抽象层要足够厚**

不要直接调用 `anthropic.messages.create`，而是通过模型适配层。适配层负责：统一消息格式翻译、内置能力差异填平。

**2. Harness 核心能力自己实现一遍**

即使 Anthropic 提供了 context editing，应用层也应有 fallback。换模型时不至于失去能力。

**3. 不依赖特定模型的隐性训练特性**

用显式 prompt 和 harness 机制保证"持续到任务完成"这类行为，不要信任模型的训练倾向。**显式的可移植，隐性的不可**。

**4. 工具设计模型无关**

工具的输入输出结构、错误码、粒度不应依赖某个模型的理解偏好。好工具对所有模型都友好。

**5. 能力探测和降级**

启动时探测模型支持什么，强模型走简路径，弱模型走重 harness 路径。

**6. 保留能力增长空间**

设计成"模型变强时可砍掉部分 harness"，而不是紧耦合。新模型发布时 Agent 自然受益，不用大改。

### 三层心智模型

```
[业务层]       你的 Agent 独特的价值
   ↑
[Harness 层]   上下文管理、工具编排、权限、观测
   ↑           （一部分逐渐下沉到平台层）
[平台/模型]    模型能力 + API 能力
```

趋势是**下面两层在吞噬上面两层的一些职责**，但：

- 业务层永远不会被吞噬
- Harness 层会变薄，但不会消失
- **跨模型兼容的需求让 harness 层在"适配复杂性"上反而变厚**

## 总结：几个值得记住的结论

1. **信噪比比长度更重要**。短不等于约束强，信息不足会让模型自由度反而变大。

2. **Prompt 的约束力来自结构而非篇幅**。角色定义要有约束性、负面清单比正面期望更有效、输出格式要机器可解析、示例要代表性强、边界情况要显式处理。

3. **Agent 的上下文管理是运行时问题**。静态 prompt 工程的经验不能直接套用。分层、压缩、外化、隔离是四个核心手法。

4. **工具设计是信噪比的源头**。工具返回值的结构决定了所有使用它的 Agent 的上下文质量。

5. **Harness 不会消失，而会重新定位**。被模型和平台吸收的是"补齐能力"的部分，留下的是"塑造身份"的部分——领域知识、流程编排、权限边界、可观测性。

6. **跨模型兼容让 harness 更复杂**。越是依赖模型内置能力，越要小心这种能力的可移植性。显式表达优于隐式信任。

7. **分层生命周期的真实实现可以参考 Claude Code**。CLAUDE.md 重新注入、tool-result clearing、compaction 的保留优先级、subagent 隔离、memory tool，这几个机制组合起来是当前较成熟的范式。
